The JSON Contract

Every search component from the server follows the same envelope:

{
  "type": "component_type",
  "data": {
    ...component-specific fields...
  }
}

type identifies which model to instantiate. data holds the actual payload. Understanding when and where that envelope is unwrapped is the key to all decoding in this module.

Some component payloads contain nested component objects that follow the same envelope convention — a component within a component. Knowing how to handle those nested levels is what this guide is about.

The Central Dispatcher: SearchComponentType

SearchComponentType is the single entry point for decoding all search components. It reads type, selects the right model, and decodes that model from the data field.

SearchComponentType.init(from decoder:) │ ├── container.requireV3(forKey: .type) → "guided_nudges" │ └── container.decode(GuidedNudgesComponentModel.self, forKey: .data) │ ▼ GuidedNudgesComponentModel.init(from decoder:) decoder now points at DATA content — the type/data envelope is gone

By the time a model's init(from:) is called, the type/data envelope has already been consumed. The model always sees only the contents of data.

This has an important consequence for testing: if you call APIv3Decoder().decode(MyModel.self, from: data) directly, the decoder starts at the root — and your model will look for its fields where "type" and "data" are, finding nothing. Always decode via SearchComponentType.self in tests to match the production dispatch path.

Decoding Patterns

Pattern 1 — Standard top-level component flat

The default pattern. The component is dispatched by SearchComponentType and its model contains flat, primitive fields. If its data has an array of items, those items are also flat objects — no per-item type/data envelope.

Example: TrendingSearchesComponentModel

SearchComponentType.init(from decoder:) │ ├── type → "trending_searches" └── decode(TrendingSearchesComponentModel.self, forKey: .data) │ ▼ TrendingSearchesComponentModel.init(from decoder:) @APIv3Model ├── title: String? → "Top searches this week" └── trendingSearches: [TrendingSearchComponentModel] │ └── each item is a flat object { "title": "...", "image": "...", "url": "..." } │ ▼ TrendingSearchComponentModel.init(from decoder:) @APIv3Model ├── title: String? ├── imageURL: URL? @CustomFieldKey("image") └── url: URL?
@APIv3Model
public struct TrendingSearchesComponentModel: SearchComponentModel, Sendable {
    @UserFacingField public var title: String?
    public var trendingSearches: [TrendingSearchComponentModel]
}

@APIv3Model
public struct TrendingSearchComponentModel: SearchComponentModel, Sendable {
    @UserFacingField public var title: String?
    @CustomFieldKey("image") public let imageURL: URL?
    public let url: URL?
}

Use this when: array items are plain JSON objects with no type/data wrapper.

Pattern 2 — Flat item arrays flat

A variant of Pattern 1 for models that are purely a list of flat items with no title or other wrapper fields. Structure is identical; called out separately because it maps 1-to-1 with a list-style component.

Example: RecentSearchListComponentModel

@APIv3Model
public struct RecentSearchListComponentModel: SearchComponentModel, Sendable {
    public var recentSearches: [RecentSearchListItemModel]
}

@APIv3Model
public struct RecentSearchListItemModel: Sendable {
    @UserFacingField public var query: String?
    public var imageUrl: String?
}

Pattern 3 — Nested component arrays via [SearchComponentType] nested

Use when array items are themselves full search components — each with their own type/data envelope. Declaring the array as [SearchComponentType] lets the dispatcher handle each element automatically, exactly as it would for a top-level component.

Example: BoxComponentModel.components, GuidedNudgesComponentModel.nudges

SearchComponentType.init(from decoder:) │ ├── type → "guided_nudges" └── decode(GuidedNudgesComponentModel.self, forKey: .data) │ ▼ GuidedNudgesComponentModel.init(from decoder:) @APIv3Model └── nudges: [SearchComponentType] │ └── for each element: { "type": "guided_nudge", "data": { ... } } │ ▼ SearchComponentType.init(from decoder:) runs again per element │ ├── type → "guided_nudge" └── decode(GuidedNudgeComponentModel.self, forKey: .data) │ ▼ GuidedNudgeComponentModel.init(from decoder:) @APIv3Model ├── title: String? @CustomFieldKey("text") ├── images: [NudgeImage] └── action: NudgeAction?
@APIv3Model
public struct GuidedNudgesComponentModel: SearchComponentModel, Sendable {
    public var nudges: [SearchComponentType]   // dispatcher handles each element
}

To access the typed model from a SearchComponentType element at the call site:

guard case .guidedNudge(let component) = nudge else { return }
let model: GuidedNudgeComponentModel = component.model

Use this when: each array element has its own type/data envelope and is already a registered SearchComponentType case.

Pattern 4 — Nested component object via private payload struct private

Use when a single field in data is itself a component-style object (type/data envelope), but you only need one or two primitive values from it — and you do not want to expose SearchComponentType in the public model API. A small private @APIv3Model struct mirrors the JSON shape and extracts what is needed.

Example: GuidedNudgesComponentModel.title

The JSON for the title field inside guided_nudges.data:

{
  "title": {
    "type": "section_title",
    "data": {
      "text": "Related searches"
    }
  }
}
GuidedNudgesComponentModel.init(from decoder:) @APIv3Model │ └── titlePayload: TitlePayload? @CustomFieldKey("title") │ decodes from { "type": "section_title", "data": { "text": "..." } } │ "type" key is ignored — not a declared property ▼ TitlePayload.init(from decoder:) @APIv3Model (private) └── data: TitleData │ decodes from { "text": "Related searches" } ▼ TitleData.init(from decoder:) @APIv3Model (private) └── text: String? @UserFacingField → "Related searches"

The public surface is a plain String? computed in an extension. The payload structs are a private implementation detail.

@APIv3Model
public struct GuidedNudgesComponentModel: SearchComponentModel, Sendable {
    @CustomFieldKey("title") private var titlePayload: TitlePayload?
    public var nudges: [SearchComponentType]

    public init(title: String?, nudges: [SearchComponentType]) {
        self.titlePayload = title.map { TitlePayload(data: TitleData(text: $0)) }
        self.nudges = nudges
    }
}

extension GuidedNudgesComponentModel {
    public var title: String? { titlePayload?.data.text }
}

private extension GuidedNudgesComponentModel {
    @APIv3Model
    struct TitlePayload {
        let data: TitleData
    }

    @APIv3Model
    struct TitleData {
        @UserFacingField let text: String?
    }
}

Use this when: a field in data is a type/data-wrapped object, but only one or two primitives are needed from it. Keeps the public model API clean without pulling in the full component type.

Choosing the Right Pattern

Situation Pattern Key type
Top-level component, all fields are primitives or flat structs Pattern 1 @APIv3Model on both container and item
Array of flat items (no per-item type/data wrapper) Pattern 1 / 2 [FlatItemModel] with @APIv3Model
Array where each item has its own type/data envelope Pattern 3 [SearchComponentType]
Single field that is a type/data component object, only need a primitive from it Pattern 4 Private @APIv3Model payload struct + computed property
Is the field an array? ├── Items are flat JSON objects (no type/data per item)? │ └── Use Pattern 1 — [FlatItemModel] with @APIv3Model │ └── Each item has a type/data envelope? └── Use Pattern 3 — [SearchComponentType] Is the field a single nested component object? ├── You need the full component model exposed publicly? │ └── SearchComponentType? with @CustomFieldKey │ └── You only need one or two primitive values from it? └── Use Pattern 4 — private payload struct + computed property

Common Pitfalls

Computed properties in the struct body The @APIv3Model macro processes every var declared in the struct body and generates a decode assignment for each one. A read-only computed property has no setter, so that assignment fails to compile. Move computed properties to an extension. 
// ✗ Fails to compile
@APIv3Model
struct MyModel {
    var action: NudgeAction?
    var actionURL: String? { action?.url }   // macro generates: actionURL = container.decodeV3(...)
}

// ✓ Correct
@APIv3Model
struct MyModel {
    var action: NudgeAction?
}

extension MyModel {
    var actionURL: String? { action?.url }
}
Decoding at the wrong JSON level in tests Calling APIv3Decoder().decode(MyModel.self, from: data) directly puts the decoder at the JSON root — the model sees "type" and "data", not its own fields. Always decode via SearchComponentType.self to match the production dispatch path. 
// ✗ Decoder at root — all fields decode as nil or empty
let model = try APIv3Decoder().decode(GuidedNudgesComponentModel.self, from: data)

// ✓ Correct — mirrors production dispatch
let component = try APIv3Decoder().decode(SearchComponentType.self, from: data)
guard case .guidedNudges(let c) = component else {
    throw TestError("Expected guidedNudges")
}
let model = c.model
Missing @CustomFieldKey when JSON key differs from Swift name @APIv3Model converts camelCase to snake_case automatically. Any other mismatch requires @CustomFieldKey explicitly. 
@CustomFieldKey("text")   @UserFacingField public var title: String?  // JSON key: "text"
@CustomFieldKey("image")  public let imageURL: URL?                    // JSON key: "image"
@CustomFieldKey("title")  private var titlePayload: TitlePayload?      // JSON key: "title"
